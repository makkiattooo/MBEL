# MBEL vs. 他のツール（エンジニア向けエディション）

ほとんどのローカライズツールは 20 年前に作られたものです。それらは現代の開発サイクルの現実を理解していません。

## 1. JSON マージ地獄
大規模なプロジェクトでは、`en.json` が戦場になります。5000 行のファイルの最後にキーを追加すると、別の開発者が同じことをするたびに Git の競合が発生します。

**MBEL の解決策**: MBEL は **名前空間によるファイルの分割** を推奨します。`auth.mbel` や `billing.mbel` で作業します。ファイルは小さく、論理的です。グローバルな巨大なオブジェクトではなく、特定のドメインファイルに触れるため、マージの競合が約 90% 削減されます。

## 2. "ICU" 構文のトラウマ
JSON 内の ICU 複数形ルールを見たことがありますか？
`{count, plural, =0{商品なし} other{{count} 個の商品}}`

文字列の中にある別の言語です。読みにくく、壊れやすいです。

**MBEL の解決策**: MBEL は **ネイティブなブロックロジック** を使用します。
```mbel
items(count) {
    [0]     => "空"
    [other] => "{count} 個の商品"
}
```
コードのように感じられ、コードのように動作します。決定論的です。

## 3. AI の「推測ゲーム」
文脈なしで AI に `Book` を送っても、それが名詞（本）なのか動詞（予約する）なのか分かりません。

**MBEL の解決策**: 第一級の **AI メタデータ**。
```mbel
@AI_Context: "ホテルの部屋を予約するボタン（動詞）"
book_btn = "今すぐ予約"
```
CLI ツールはこの文脈を LLM に渡し、人間の修正なしで 99.9% の翻訳精度を保証します。
