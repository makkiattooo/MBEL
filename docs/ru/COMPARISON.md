# MBEL vs. Остальной мир (Версия для инженеров)

Большинство инструментов локализации были созданы 20 лет назад. Они не понимают реальности современных циклов разработки.

## 1. Ад конфликтов слияния JSON
В масштабных проектах `en.json` становится полем битвы. Добавление одного ключа в конец 5000-строчного файла приводит к конфликту Git почти каждый раз, когда другой разработчик делает то же самое.

**Решение MBEL**: MBEL поощряет **файлы с пространствами имен**. Вы работаете над `auth.mbel` или `billing.mbel`. Файлы небольшие и логичные. Конфликты слияния сокращаются на ~90%.

## 2. Травма синтаксиса "ICU"
Вы видели правила множественного числа ICU в JSON?
`{count, plural, =0{нет товаров} one{1 товар} few{# товара} many{# товаров} other{# товаров}}`

Это язык внутри строки. Это нечитабельно и хрупко.

**Решение MBEL**: MBEL использует **нативную логику блоков**.
```mbel
items(count) {
    [0]     => "Пусто"
    [one]   => "1 товар"
    [few]   => "{count} товара"
    [many]  => "{count} товаров"
}
```
Это похоже на код. Это работает как код. Это детерминировано.

## 3. Игра в "угадайку" для ИИ
Без контекста ИИ не знает, является ли `Книга` существительным или глаголом (забронировать).

**Решение MBEL**: **ИИ-метаданные** первого класса.
```mbel
@AI_Context: "Кнопка для бронирования гостиничного номера (Глагол)"
book_btn = "Забронировать сейчас"
```
Наши инструменты CLI передают этот контекст в LLM, гарантируя точность перевода 99,9%.
